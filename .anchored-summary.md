## Goal
- Build and maintain a sports schedule directory site matching ibuffstreams.app design with Astro + Tailwind.

## Constraints & Preferences
- Dark theme (buff-dark #121212, buff-darker #1f1f1f, buff-card #121212) with teal accent (#0a93ae) and live‑red (#bf0c3e).
- Layout: max‑w‑7xl, 2‑column grid (lg:grid‑cols‑3) on main pages, content col‑span‑2 + sidebar col‑span‑1.
- Header: title left, nav right, hamburger (<sm) toggles dropdown.
- SportGrid: horizontal scroll, gap‑5, py‑1, mr‑3, 38x38 icons, scroll‑smooth.
- Match‑card separators: divide‑dashed divide‑buff‑border (#404040).
- Live timer/playing time red; countdown/upcoming/calendar‑active teal (text‑buff‑gold / bg‑buff‑gold).
- Sport section header icons: no border‑radius (not rounded‑full).
- Stream sources fetched client‑side from /api/proxy/stream/{source}/{id}; built fallback to "No streams available".
- Countdown reads target from data‑target attribute, not Astro template expressions in <script>.
- Scripts inside named slots (not page‑level) to keep them inside <body>.
- To pass Astro server data to client scripts, use `<script is:inline set:html={...}>` with backtick template literals containing `${JSON.stringify(...)}` sub‑expressions.

## Key Decisions
- `<script is:inline set:html={...}>` with backtick template literals is the correct way to embed server data into client scripts in Astro. Plain `<script>` tags do NOT process `{...}` template expressions; `is:inline` without `set:html` also does not process them. Only `set:html` with a template literal expression works.
- Scripts placed inside `<div slot="content">` instead of page level to keep them inside <body>.
- Countdown uses data-target attribute to pass server value to client script (avoids Astro template issues in <script> tags).
